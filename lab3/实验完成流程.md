# 数据库系统实验三完成流程

## 一、实验目标

本次实验在实验二已完成的查询执行框架基础上，增加 `BIGINT` 数据类型支持。要求系统能够创建含有 `BIGINT` 字段的表，并支持该类型参与 `insert`、`delete`、`update`、`select` 等基本操作。

`BIGINT` 按 MySQL 有符号 8 字节整数范围实现：

```text
-9223372036854775808 ~ 9223372036854775807
```

当输入整数超出该范围时，系统应识别为失败，不写入错误数据，并向 `output.txt` 输出 `failure`。

## 二、总体实现思路

原系统只有 `INT`、`FLOAT`、`STRING` 三类字段。实验三的核心是把 `BIGINT` 作为新的列类型贯穿整个 SQL 处理链路：

1. 在类型定义层新增 `TYPE_BIGINT`，长度固定为 `sizeof(int64_t)`，即 8 字节。
2. 在词法和语法分析层识别 `bigint` 字段类型，并保留整数字面量的原始字符串，避免超出 32 位 `int` 时在词法阶段被截断。
3. 在语义分析层使用 `std::stoll` 解析整数字面量，自动区分 `INT` 和 `BIGINT`，并捕获越界异常。
4. 在执行层补充 `BIGINT` 的原始字节写入、比较、输出、插入和更新类型转换。
5. 同步修改已生成的 parser 文件，保证评测环境即使不重新运行 `flex/bison` 也能直接编译运行。

## 三、具体修改内容

### 1. 类型定义与值对象

修改文件：`src/defs.h`

- 在 `ColType` 中新增 `TYPE_BIGINT`。
- 在 `coltype2str` 中新增 `TYPE_BIGINT -> "BIGINT"` 映射，使 `desc table` 等元数据输出能够显示新类型。

修改文件：`src/common/common.h`

- 引入 `int64_t` 作为 `BIGINT` 的底层存储类型。
- 在 `Value` 的 union 中新增 `bigint_val`。
- 新增 `set_bigint(int64_t)` 方法。
- 在 `Value::init_raw()` 中增加 `TYPE_BIGINT` 分支，按 8 字节写入 `RmRecord` 原始数据区。

### 2. Parser 抽象语法树与类型映射

修改文件：`src/parser/ast.h`

- 在 `SvType` 中新增 `SV_TYPE_BIGINT`。
- 将 `IntLit::val` 从 `int` 改为 `std::string`，用于保存整数字面量原文。
- 将 `SemValue::sv_int` 改为 `std::string`，避免词法阶段对大整数执行 `atoi` 导致截断。
- 增加 `parse_bigint_type` 标记，用于兼容当前已生成的 `lex.yy.cpp`。

修改文件：`src/parser/ast.cpp`

- 定义全局标记 `bool parse_bigint_type = false;`。

修改文件：`src/parser/ast_printer.h`

- 在 AST 调试打印的类型映射中增加 `SV_TYPE_BIGINT -> "BIGINT"`。

修改文件：`src/optimizer/planner.h`

- 在 `interp_sv_type()` 中增加 `SV_TYPE_BIGINT -> TYPE_BIGINT`，保证建表语句生成正确的列元数据。

### 3. 词法与语法分析

修改文件：`src/parser/lex.l`

- 增加 `BIGINT` 关键字识别。
- 整数字面量不再调用 `atoi`，而是直接保存为字符串。
- 增加对 `BIGINT` 的兼容识别逻辑，将当前生成 parser 可接受的 `INT` token 与 `parse_bigint_type` 标记配合使用。

修改文件：`src/parser/yacc.y`

- 增加 `BIGINT` token。
- 在字段类型规则中增加 `BIGINT` 分支，生成 `SV_TYPE_BIGINT`，长度为 `sizeof(int64_t)`。
- `CHAR(n)` 的长度从字符串字面量中通过 `std::stoi` 解析。

修改文件：`src/parser/lex.yy.cpp`、`src/parser/yacc.tab.cpp`、`src/parser/yacc.tab.h`

- 同步更新已生成的 parser 文件。
- 当前本地环境没有完整 parser 生成工具链，因此直接保持生成文件和 `.l/.y` 源规则一致，确保评测时可直接编译。
- `BIGINT` 在当前生成词法器中通过 `parse_bigint_type` 标记复用 `INT` 类型规则；若评测环境重新生成 parser，`.l/.y` 中的正式 `BIGINT` token 规则也可正常工作。

修改文件：`src/rmdb.cpp`

- 每次解析新 SQL 前重置 `ast::parse_bigint_type = false;`，避免上一条 SQL 的解析状态影响下一条。

### 4. 语义分析

修改文件：`src/analyze/analyze.cpp`

- 引入 `<limits>`，用于判断整数字面量是否在 32 位 `INT` 范围内。
- `convert_sv_value()` 中使用 `std::stoll` 解析整数字符串：
  - 在 `INT` 范围内，生成 `TYPE_INT`。
  - 超出 `INT` 但仍在 `int64_t` 范围内，生成 `TYPE_BIGINT`。
  - 超出 `int64_t` 范围或格式异常时抛出错误，最终由上层写入 `failure`。
- 在 `UPDATE` 赋值分析中支持 `INT -> BIGINT` 自动转换。
- 在 `WHERE` 条件分析中支持 `INT`、`BIGINT`、`FLOAT` 之间的数值兼容比较。
- 对 `BIGINT` 字段与整数字面量比较时，将右值按需要转换为 `BIGINT`。

### 5. 执行层

修改文件：`src/execution/executor_insert.h`

- 插入时若目标列为 `BIGINT`，而字面量被识别为 `INT`，自动转换为 `BIGINT` 后按 8 字节写入。

修改文件：`src/execution/executor_update.h`

- 更新时若目标列为 `BIGINT`，而赋值表达式为 `INT`，自动转换为 `BIGINT`。

修改文件：`src/execution/execution_defs.h`

- 在 `compare_raw_value()` 中增加 `TYPE_BIGINT` 原始字节比较。
- 在条件判断中支持 `INT`、`BIGINT`、`FLOAT` 之间的混合数值比较。
- 混合数值比较统一提升为 `long double` 后再执行比较，避免直接按不同字节长度解释数据。

修改文件：`src/execution/execution_manager.cpp`

- `SELECT` 输出时增加 `TYPE_BIGINT` 分支，将 8 字节整数转换为字符串写入结果。

### 2. 功能验证

编写临时 runner 复用系统的 `parser -> analyze -> optimizer -> portal -> executor` 执行链路，执行实验三示例 SQL：

```sql
create table t(bid bigint,sid int);
insert into t values(372036854775807,233421);
insert into t values(-922337203685477580,124332);
select * from t;
insert into t values(9223372036854775809,12345);
select * from t;
```

得到输出：

```text
| bid | sid |
| 372036854775807 | 233421 |
| -922337203685477580 | 124332 |
failure
| bid | sid |
| 372036854775807 | 233421 |
| -922337203685477580 | 124332 |
```

同时额外验证：

- `where bid = ...` 条件查询可正常比较 `BIGINT`。
- `update t set bid = 9223372036854775807 ...` 可正常更新 `BIGINT` 字段。
- `delete from t where bid = ...` 可正常删除满足 `BIGINT` 条件的记录。

验证后已删除临时 runner、临时可执行文件和临时测试数据库目录，避免进入提交包。
