# Lab6 实验完成流程

## 1. 实验目标
- 根据 lab6.md 要求，在 RMDB 中实现 SQL 聚合函数 SUM、MAX、MIN、COUNT
- 支持 `COUNT(*)`、`COUNT(col)`、`SUM(col)`、`MAX(col)`、`MIN(col)`
- 支持 `AS` 别名语法
- 支持与 `WHERE` 条件组合使用
- 浮点数保留 6 位小数，整数不显示小数

## 2. 需求拆解
- 子任务 1：词法分析器添加 SUM/MAX/MIN/COUNT/AS 关键字
- 子任务 2：语法分析器添加聚合函数表达式和 SELECT 聚合查询语法规则
- 子任务 3：AST 中新增 AggExpr 节点和 AggType 枚举
- 子任务 4：语义分析器解析聚合表达式，确定结果类型和别名
- 子任务 5：查询计划层新增 AggregatePlan
- 子任务 6：执行引擎新增 AggregateExecutor（阻塞式聚合算子）
- 子任务 7：Portal 层适配 AggregatePlan 到 AggregateExecutor 的转换

## 3. 实现思路

### 3.1 聚合查询语法解析
- 原有流程：`SELECT selector FROM ...` 中 `selector` 仅支持 `*` 或列列表
- 实现方案：新增独立的 `agg_list` 语法规则，支持 `SUM(col) as alias` 形式
- 关键数据结构：`ast::AggExpr` 包含 `agg_type`、`is_count_star`、`col`、`alias`
- SELECT 规则新增分支：`SELECT agg_list FROM tableList optWhereClause opt_order_clause`

### 3.2 语义分析
- 原有流程：SelectStmt 的 `cols`（TabCol 列表）用于投影
- 实现方案：新增 `AggSel` 结构体描述聚合选择，存储在 `Query.agg_sels` 中
- 类型推导规则：
  - COUNT(*), COUNT(col) → TYPE_INT
  - SUM(col) → 参数列类型（仅支持 int/float）
  - MAX(col), MIN(col) → 参数列类型（支持 int/float/char）

### 3.3 阻塞式聚合算子
- 原有流程：所有执行器都是火山模型（迭代式 pull）
- 实现方案：AggregateExecutor 采用阻塞式设计，在 `beginTuple()` 中消费所有子算子的行并计算结果，然后只输出一行
- 聚合逻辑：
  - COUNT：行计数
  - SUM：int 累加到 long long，float 累加到 double
  - MAX/MIN：逐行比较取极值，支持 int/float/bigint/string

### 3.4 输出格式
- `select_from()` 通过 `cols()` 获取 ColMeta 格式化输出
- AggregateExecutor 返回合成的 ColMeta（name=别名, type=结果类型）
- float 类型通过 `std::to_string()` 输出，默认 6 位小数，符合题目要求

### `rmdb/src/common/common.h`
- 新增 `AggSel` 结构体（agg_type、is_count_star、col、alias、agg_col_type）
- 新增 `#include "parser/ast.h"` 以使用 `ast::AggType`
- 【本次修复】`AggSel` 新增 `int agg_col_len = 0` 字段，记录聚合结果列的字节宽度
  - 修复前聚合输出列宽由硬编码的 `get_type_len()` 决定，对 TYPE_STRING 固定返回 8 字节
  - 导致 char(N>8) 列的 MAX/MIN 结果被截断为 8 字节（隐藏测试点 aggregate_test2 失败，得分 5.00）

### `rmdb/src/analyze/analyze.h` / `analyze.cpp`
- `Query` 类新增 `agg_sels` 字段
- `do_analyze` 方法新增聚合表达式处理逻辑：解析参数列、推导结果类型、设置别名
- 【本次修复】在聚合分支中为 `agg_sel.agg_col_len` 赋值：
  - COUNT(*) / COUNT(col) → `sizeof(int)`（结果恒为整型）
  - SUM(col) / MAX(col) / MIN(col) → `col_meta->len`（参数列的真实字节宽度，如 char(20)→20）

### `rmdb/src/optimizer/plan.h`
- `PlanTag` 枚举新增 `T_Aggregate`
- 新增 `AggregatePlan` 类（subplan_、aggs_）

### `rmdb/src/optimizer/planner.cpp`
- `generate_select_plan` 方法修改：检测 `agg_sels` 非空时创建 `AggregatePlan`，否则创建 `ProjectionPlan`

### `rmdb/src/execution/executor_aggregate.h`（新建）
- `AggregateExecutor` 类：阻塞式聚合执行器
- `beginTuple()` 消费所有子行，计算 SUM/MAX/MIN/COUNT
- `Next()` 返回单行聚合结果
- `cols()` 返回合成 ColMeta（别名作为列名）
- 【本次修复】
  - 构造函数中输出列宽改为 `col.len = agg.agg_col_len`（取源列真实宽度），替代旧 `get_type_len(agg.agg_col_type)`
  - 删除错误的 `static get_type_len(ColType)` 辅助函数（其 `default` 分支对 TYPE_STRING 返回 `sizeof(long long)=8`，正是截断 bug 的根因）
  - MAX/MIN 的 string 累积逻辑不变（仍用 `src_col.len` 读取源值、按 `cols_[i].len` 写入），由于 analyze 已令 `agg_col_len = col_meta->len`，二者一致，char(N) 结果不再被截断

### `rmdb/src/portal.h`
- 新增 `#include "execution/executor_aggregate.h"`
- `start` 方法 T_select 分支：检测 AggregatePlan，构建别名作为 sel_cols
- `convert_plan_executor` 方法：新增 AggregatePlan → AggregateExecutor 转换

## 5. 测试与结果

### 5.1 构建
- 命令：`cd rmdb/build && cmake .. && make -j$(nproc)`（在 WSL 中执行）
- 结果：通过。无 aggregate 相关编译警告/错误（仓库原有 39 条无关警告）。产物 `build/bin/rmdb`。

### 5.2 官方测试（lab6.md 全部 3 个测试点）
- 测试方式：rmdb 为 TCP socket 服务器（端口 8765，不读 stdin）。启动后用 Python socket 客户端发送 SQL，读取数据库目录下的 `output.txt`（评分系统读取的文件）逐行比对。
- 关键：`output.txt` 写入数据库目录 `build/<db_name>/output.txt`，格式为 `| <表头> |` + `| <值> |`。
- 测试点1 SUM：
  - `SELECT SUM(id) AS sum_id` → `| 9 |` ✓
  - `SELECT SUM(val) AS sum_val` → `| 20.000000 |` ✓
- 测试点2 MAX/MIN：
  - `SELECT MAX(id) AS max_id` → `| 5 |` ✓
  - `SELECT MIN(val) AS min_val` → `| 4.500000 |` ✓
- 测试点3 COUNT：
  - `SELECT COUNT(*) AS count_row` → `| 3 |` ✓
  - `SELECT COUNT(id) AS count_id` → `| 3 |` ✓
  - `SELECT COUNT(name) AS count_name WHERE val=2.0` → `| 2 |` ✓
- 结果：7 个断言全部与 lab6.md 标准答案逐字符匹配（20/20 行匹配，0 mismatch）。

### 5.3 关键自测：char(N>8) MAX/MIN（复现并修复 aggregate_test2）
- 背景：外部评分首次提交得分 5.00，隐藏测试点 `aggregate_test2` 失败。lab6.md 公布用例全是 char(8)，旧 bug 下 char(8) 恰好 8 字节容下"碰巧"通过，无法触发问题。
- 根因定位：旧版 `get_type_len()` 对 `TYPE_STRING` 返回固定 8 字节，导致 char(N>8) 的 MAX/MIN 输出列宽被强制为 8，`QlManager::select_from` 打印 `std::string(rec_buf, col.len)` 时截断到 8 字节，与标准答案不符。
- 针对性自测（修复后，结果对用户可见）：
  - char(20) 表，插 `'abcdefghij'`(10)、`'xyz'`(3)、`'abcdefghijklmnopqrst'`(20)：
    - `MAX(name)` → `| xyz |` ✓（字典序最大，完整未截断）
    - `MIN(name)` → `| abcdefghij |` ✓（完整 10 字符；旧 bug 会截成 `abcdefgh`）
  - char(20) 表，插 `'abcdefghijkl'`(12)：
    - `MAX(name)` → `| abcdefghijkl |` ✓（完整 12 字符；旧 bug 会截成 `abcdefgh`）
- 结论：`agg_col_len` 修复对症，char(N>8) 截断 bug 已消除。
