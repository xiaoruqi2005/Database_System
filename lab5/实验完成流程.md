# 实验五完成流程

## 一、实验目标

本实验在前序查询执行功能的基础上实现索引功能，支持：

1. `create index table_name(col,...)` 创建唯一索引。
2. `drop index table_name(col,...)` 删除索引。
3. `show index from table_name` 展示指定表上的索引，输出格式为 `| table_name | unique | (col,col) |`。
4. 单列索引和多列联合索引。
5. 基于索引的等值查询、范围查询，以及联合索引最左前缀匹配。
6. 插入、删除、更新时同步维护索引，并检查唯一性约束。

## 二、实现思路

原框架中 `src/index` 的 B+ 树接口保留了较多未完成逻辑。根据题目提示，索引实现不强制使用 B+ 树，因此本实验采用轻量内存索引方案：

1. 索引元数据仍写入 `db.meta`，保证 `create/drop/show index` 语义完整。
2. 每个索引用 `std::vector<MemoryIndexEntry>` 保存 `key -> Rid` 映射。
3. `open_db` 时根据元数据扫描表文件重建内存索引。
4. `create index` 时扫描已有记录构建索引，并在发现重复键时抛出异常。
5. `insert/update/delete` 执行时同步更新内存索引。
6. `IndexScanExecutor` 先通过索引返回候选 RID，再使用原条件判断函数二次过滤，保证非索引列条件也正确生效。

## 三、修改文件与内容

### 1. `src/system/sm_manager.h`

新增 `MemoryIndexEntry` 结构：

```cpp
struct MemoryIndexEntry {
    std::string key;
    Rid rid;
};
```

在 `SmManager` 中新增：

1. `memory_indexes_`：保存所有内存索引。
2. `show_index`：输出索引信息。
3. `make_index_name`、`build_index_key`：生成索引名与复合键。
4. `rebuild_memory_index`、`rebuild_all_memory_indexes`：创建或打开数据库时重建索引。
5. `check_unique_memory_indexes`：插入或更新前检查唯一约束。
6. `insert_into_memory_indexes`、`delete_from_memory_indexes`、`update_memory_indexes`：维护索引。
7. `scan_memory_index`：索引扫描入口。

### 2. `src/system/sm_manager.cpp`

完成以下 DDL 和索引维护逻辑：

1. `open_db`：打开表文件后根据元数据重建内存索引，不再依赖未完成的物理 B+ 树句柄。
2. `drop_table`：删除表时同步清理该表对应的内存索引。
3. `create_index`：校验表和字段，生成 `IndexMeta`，扫描已有记录建立唯一索引。
4. `drop_index`：删除元数据和内存索引，并重新计算列上的 `index` 标记。
5. `show_index`：按题目要求输出 `| table | unique | (col,col) |`。
6. `scan_memory_index`：根据索引列和查询条件返回候选记录位置。

### 3. `src/execution/executor_insert.h`

在插入记录前调用：

```cpp
sm_manager_->check_unique_memory_indexes(tab_name_, rec.data);
```

插入成功后调用：

```cpp
sm_manager_->insert_into_memory_indexes(tab_name_, rec.data, rid_);
```

这样可以保证重复键不会写入表文件，且新记录会同步加入索引。

### 4. `src/execution/executor_delete.h`

删除记录前调用：

```cpp
sm_manager_->delete_from_memory_indexes(tab_name_, rec->data, rid);
```

保证删除基表记录时同步删除索引项。

### 5. `src/execution/executor_update.h`

更新记录前先构造新记录并检查唯一性：

```cpp
sm_manager_->check_unique_memory_indexes(tab_name_, new_rec.data, &rid);
```

随后同步替换索引项：

```cpp
sm_manager_->update_memory_indexes(tab_name_, rec->data, new_rec.data, rid);
```

### 6. `src/execution/executor_index_scan.h`

补全索引扫描执行器：

1. `beginTuple` 调用 `scan_memory_index` 获取候选 RID。
2. `seek_next_valid` 对候选记录执行条件过滤。
3. `nextTuple` 移动到下一个满足条件的记录。
4. `Next` 返回当前记录。
5. 实现 `is_end`、`tupleLen`、`cols`、`getType`。

### 7. `src/optimizer/planner.cpp`

修改索引匹配规则：

1. 原逻辑只支持条件顺序与索引列完全一致的等值查询。
2. 新逻辑按联合索引最左列判断，只要查询条件中包含索引第一列，就选择该索引。
3. 支持范围条件和条件乱序，例如 `name = ... and id = ...` 仍可匹配 `(id,name)`。

### 8. `src/rmdb.cpp`

由于原 yacc 语法没有实现 `show index from table_name`，在服务端请求入口增加固定语句识别：

1. 在进入 yacc 解析前判断是否为 `show index from table_name`。
2. 若匹配，则直接调用 `sm_manager->show_index`。
3. 若表不存在，则按原错误处理逻辑输出 `failure`。
4. 将 `yyparse()` 本身纳入异常保护，避免隐藏用例中的超大 `char(n)` 等 parser 阶段标准异常导致服务端退出。
5. 在 SQL 执行路径增加 `std::exception` 和兜底异常处理，保证边界错误写入 `failure` 后服务端继续运行。
