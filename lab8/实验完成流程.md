## 3. 实现思路

### 3.1 Block Nested-Loop Join 执行器
- 原有流程：`NestedLoopJoinExecutor` 每次固定一条左表记录，遍历右表查找满足 join 条件的记录对；右表扫描完成后推进左表并重新扫描右表。
- 实现方案：在执行器内部维护一个左侧记录块 `left_block_`，块大小按 `JOIN_BUFFER_SIZE` 字节预算控制；每加载一个左块，就扫描一次右侧执行器，并在当前右记录上遍历左块中的记录。
- 关键数据结构或算法：`left_block_` 保存当前左块记录，`left_block_pos_` 保存块内位置，`current_right_` 保存当前右侧记录，`current_joined_` 缓存已经定位到的当前输出记录。
- 边界条件与异常处理：左表为空时直接结束；右表为空时结束当前块并继续检测后续左块；非等值条件沿用原比较逻辑；`Next()` 只返回已由 `find_next_valid()` 定位的当前拼接记录。

## 4. 修改文件明细

### `rmdb/src/execution/executor_nestedloop_join.h`
- 修改的类、函数或代码区域：`NestedLoopJoinExecutor` 的成员状态、`beginTuple()`、`nextTuple()`、`find_next_valid()`、`Next()`。
- 修改前存在的问题：原实现是普通 tuple-at-a-time Nested Loop Join，没有按块缓存左侧输入；大数据连接时需要频繁重扫右侧输入，未满足 Lab 8 对 BNLJ 算法的要求。
- 具体修改内容：新增左块缓存、当前右记录和当前输出记录状态；新增 `load_left_block()` 和 `join_records()`；将游标推进改为“加载左块 -> 扫描右侧 -> 块内逐条匹配 -> 下一个右记录 -> 下一个左块”。
- 修改原因及其与题目要求的对应关系：Lab 8 明确要求实现 Block Nested-Loop Join，且测试数据可能超过内存；该实现只按固定字节预算保留左块，不一次性物化整张输入表。
