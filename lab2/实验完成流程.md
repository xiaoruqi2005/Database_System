# 实验二完成流程

## 一、实验目标

实验二在实验一存储管理功能的基础上，继续实现基础 SQL 功能，使系统支持：

- 元数据管理与 DDL：`create table`、`drop table`、`show tables`
- DML：`insert`、`update`、`delete`
- DQL：单表查询、条件查询、投影查询、双表连接查询
- 非法 SQL 或语义错误输出 `failure`

实验测试点包括：

- `basic_query_test1`：建表、删表、显示表
- `basic_query_test2`：单表插入与条件查询
- `basic_query_test3`：单表更新与条件查询
- `basic_query_test4`：单表删除与条件查询
- `basic_query_test5`：连接查询

## 二、总体实现思路

实验二的实现路线是沿着 SQL 执行链路补齐缺失模块：

1. `parser` 将 SQL 转为 AST，框架已提供。
2. `analyze` 做语义分析，检查表、列、类型，并生成 `Query`。
3. `planner` 生成逻辑/物理执行计划。
4. `portal` 将计划转为执行器树。
5. `execution` 执行扫描、投影、连接、更新和删除。
6. `system` 负责数据库和表元数据管理。
7. `transaction` 提供最小可用事务对象，避免服务端单语句事务路径空指针。

基础测试不涉及索引功能，因此索引相关复杂逻辑没有作为实验二重点实现；执行路径默认使用顺序扫描和嵌套循环连接。

### 2. `src/system/sm_manager.cpp`

实现数据库元数据管理和表删除。

修改点：

- 实现 `SmManager::open_db`
  - 检查数据库目录是否存在。
  - `chdir` 进入数据库目录。

  - 关闭该表的数据文件句柄。
  - 删除该表关联的索引文件。

  - 刷新 `db.meta`。

## 3. `src/analyze/analyze.cpp`

补充语义分析。

修改点：

- `SelectStmt`
  - 检查 `from` 中所有表是否存在。
  - 对投影列做列名解析。
  - 当列名未指定表名时，根据所有候选表推断表名。
  - 若列名在多个表中出现，抛出 `AmbiguousColumnError`。
  - 若列不存在，抛出 `ColumnNotFoundError`。

- `UpdateStmt`
  - 检查目标表是否存在。
  - 解析 `set` 子句为 `SetClause`。
  - 检查被更新列是否存在。
  - 检查赋值类型是否兼容。
  - 支持 `int` 字面量赋给 `float` 列，会自动转为 float。
  - 解析并检查 where 条件。

- `DeleteStmt`
  - 检查目标表是否存在。
  - 解析并检查 where 条件。

- `InsertStmt`
  - 检查目标表是否存在。
  - 将 AST 中的字面量转为执行层 `Value`。

- `check_column`
  - 补充显式表名情况下的列存在性检查。

- `check_clause`
  - 检查 where 条件左右两侧类型。
  - 允许 `int` 和 `float` 之间进行数值比较。
  - 当 float 列与 int 字面量比较时，将 int 字面量转换为 float。

## 4. `src/execution/execution_defs.h`

增加执行期条件比较辅助函数。

修改点：

- 增加 `compare_raw_value`
  - 支持 int、float、string 三种类型的原始字节比较。
  - float 比较使用 `1e-6` 容差。

- 增加 `compare_result`
  - 根据 `CompOp` 判断比较结果是否满足条件。

- 增加 `find_col`
  - 在执行器输出列集合中定位目标列元数据。

- 增加 `eval_condition`
  - 根据 `Condition` 对一条记录执行条件判断。
  - 支持列和值比较，也支持列和列比较。
  - 支持 int/float 数值兼容比较。

- 增加 `eval_conditions`
  - 对多个 where 条件做 AND 判断。

这些辅助函数被顺序扫描和连接执行器复用。

## 5. `src/execution/executor_seq_scan.h`

实现顺序扫描执行器。

修改点：

- 增加 `seek_next_valid`
  - 从当前扫描位置开始查找下一条满足条件的记录。
  - 使用 `RmScan` 遍历表记录。
  - 使用 `eval_conditions` 判断 where 条件。

- 实现 `beginTuple`
  - 初始化 `RmScan`。
  - 定位第一条满足条件的记录。

- 实现 `nextTuple`
  - 推进扫描器。
  - 定位下一条满足条件的记录。

- 实现 `Next`
  - 返回当前 rid 对应的记录。

- 实现 `is_end`
  - 判断扫描是否结束。

- 实现 `tupleLen`、`cols`、`getType`
  - 为上层投影和连接执行器提供记录长度和列元数据。

## 6. `src/execution/executor_projection.h`

实现投影执行器。

修改点：

- 构造函数中根据选择列 `sel_cols` 找到子执行器输出列位置。

- 实现 `beginTuple` 和 `nextTuple`，直接转发给子执行器。

## 7. `src/execution/executor_nestedloop_join.h`

实现嵌套循环连接执行器。

修改点：

- 合并左右子执行器的列元数据。
- 右表列 offset 增加左表记录长度。
- 实现连接记录拼接：
  - 左记录内容放在前半部分。
  - 右记录内容放在后半部分。

- 实现 `beginTuple`
  - 初始化左执行器。
  - 将右执行器结果物化到 `right_records_`。
  - 从右侧结果末尾开始遍历，使 `select * from t, d` 的输出顺序与题面标准一致。

- 实现 `seek_next_valid`
  - 遍历左记录和右记录组合。
  - 对连接后的记录执行连接条件判断。
  - 找到满足条件的组合后暂停。

- 实现 `nextTuple`
  - 推进右侧物化结果下标。
  - 当前左记录匹配完后进入下一条左记录。

- 实现 `Next`
  - 返回当前连接结果。

- 实现 `is_end`、`tupleLen`、`cols`、`getType`。

特别说明：

题面 `basic_query_test5` 中 `select * from t, d;` 的标准答案要求右表后插入记录先输出，因此连接执行器对右侧结果采用倒序遍历。

## 四、关键实现细节

### 1. 条件判断统一处理

where 条件在语义分析阶段完成列名解析和类型检查，在执行阶段由 `eval_conditions` 统一判断。这样扫描和连接可以复用同一套逻辑。

### 2. int 与 float 兼容

题面示例包含如下 SQL：

```sql
select score,name,id from grade where score > 90;
update grade set name = 'test' ,id = -1,score = 0 where name = 'test' and score > 90;
```

其中 `score` 是 float，但字面量 `90` 和 `0` 是 int。因此实现中允许 int 和 float 兼容，并在需要时将 int 转为 float。

### 3. 投影列 offset 重排

投影执行器输出的是新记录，因此不能沿用原表 offset。实现中按选择列顺序重新计算 offset，再从子记录拷贝对应字节。

### 4. 连接输出顺序

为了匹配题面标准答案，连接执行器保持左表顺序不变，对右表物化结果倒序遍历。这样 `select * from t, d;` 的输出为：

```text
| 1 | aaa | 23456 | 2 |
| 1 | aaa | 12345 | 1 |
...
```

同时带连接条件的查询仍然按左表顺序输出匹配结果。

### 5. 输出文件

查询输出写入当前数据库目录下的 `output.txt`。`open_db` 时会清空旧输出，后续每条 SQL 追加结果。
