# AGENTS.md

## 1. 项目目标

本仓库包含数据库系统实现课程的 10 个实验，目录通常为：

```text
项目根目录/
├── AGENTS.md
├── docs/                  # 课程说明文档、项目结构文档和图片资源，不进入提交压缩包
├── rmdb/                  # 当前主开发源码目录，始终保存最新实现
│   ├── CMakeLists.txt
│   ├── deps/
│   ├── rmdb_client/
│   └── src/
├── lab1/
│   ├── db2023
│   ├── lab1.md
│   ├── lab1.zip               # 最终提交压缩包
│   └── 实验完成流程.md
├── lab2/
│   ├── db2023
│   ├── lab2.md
│   ├── lab2/                  # 完成 Lab2 后从 rmdb/ 同步得到的交付源码目录
│   ├── lab2.zip
│   └── 实验完成流程.md
└── ...
```

你的任务是：**严格依据目标实验的 `labx.md`，在 `rmdb/` 中完成当前最新源码实现；实验完成后同步到 `labx/labx/`，再完成实验记录、压缩交付、Git 提交并推送。**

除非用户明确指定多个实验，否则一次只处理一个 `labx`，不得顺手修改其他实验。

## 2. 最高优先级规则

按以下优先级理解任务，发生冲突时以前者为准：

1. 当前目标实验的 `labx/labx.md`；
2. 课程提供的测试、已有接口、头文件声明、代码注释和错误类型定义；
3. 当前仓库中的构建脚本、`docs/` 中的 README 和项目结构文档；
4. `rmdb/` 中当前已有实现，作为后续实验的代码基线；
5. `lab1/` 中已经完成的内容，仅作为实验报告和压缩交付格式的范例；
6. 本文件中的通用规则。

必须遵守：

* 不得遗漏 `labx.md` 中列出的任何接口、功能、异常行为或测试点。
* 不得修改已有公开接口的函数名、参数、返回值和语义，除非 `labx.md` 明确允许。
* 不得删除题目已有的数据结构、成员变量、测试代码或必要功能。
* 不得通过硬编码测试答案、伪造输出、跳过测试、放宽断言等方式过关。
* 不得修改官方测试来掩盖实现错误；可以新增自测，但不能让新增测试替代官方测试。
* 不得只实现示例用例；需要考虑边界条件、错误路径、资源释放和隐藏测试。
* 保持与项目现有 C++ 标准、代码风格、异常体系和构建方式一致。
* 优先做最小且完整的修改，避免与当前实验无关的重构。

## 3. 开始工作前的检查

开始目标实验 `labx` 前，依次执行并记录：

```bash
pwd
git status --short
git branch --show-current
git remote -v
```

然后完成以下检查：

1. 确认目标目录是 `labx/`，说明文档是 `labx/labx.md`，主开发源码目录是 `rmdb/`。
2. 完整阅读 `labx/labx.md`，提取：

   * 必须实现的接口；
   * 必须支持的行为；
   * 非法输入和异常要求；
   * 测试入口、测试点和期待输出；
   * 禁止修改的接口或数据结构；
   * 前置实验依赖。
3. 阅读 `docs/` 中的 README 和项目结构文档，以及 `rmdb/` 中的 CMakeLists、相关头文件、实现文件、测试和注释。
4. 搜索待实现位置，但不能只依赖 `TODO`：

```bash
rg -n "TODO|FIXME|YOUR CODE|IMPLEMENT|未实现|throw|assert\(false\)" rmdb
```

5. 查看 `lab1/` 的成品结构，重点确认：

   * `实验完成流程.md` 的组织方式；
   * `lab1.zip` 内部的路径层级；
   * 哪些文件被排除在压缩包之外。

`lab1` 只是交付格式范例。不得因为 Lab1 的实现方式与当前实验不同，就机械复制其业务逻辑。

若工作区已有其他成员的未提交修改：

* 不覆盖、不回滚、不格式化这些无关修改；
* 不执行 `git reset --hard`、`git clean -fd`、强制 checkout 等破坏性命令；
* 仅修改 `rmdb/` 中当前实验所需文件，以及最终交付需要的 `labx/` 文件；
* 提交时只暂存 `rmdb/` 和当前实验相关文件。

## 4. 实现流程

### 4.1 建立需求清单

在编码前，先在内部形成逐项清单，将 `labx.md` 中每个要求映射到：

* 相关模块；
* 相关文件；
* 相关函数或类；
* 对应测试；
* 关键边界条件。

不得在未读完题目和相关调用链时直接批量填写 TODO。

### 4.2 理解调用链

对每个待实现接口，至少检查：

* 调用者如何使用该接口；
* 输入参数的合法范围和所有权；
* 返回值或异常如何被上层处理；
* 是否涉及锁、页固定计数、脏页、文件描述符、内存所有权、事务或日志；
* 是否存在与之配套的反向操作和资源释放；
* 是否会影响前置实验功能。

### 4.3 编码原则

* 复用仓库已有工具、异常类和辅助函数，不重复造功能相同的机制。
* 新增成员变量或辅助函数前，确认不会破坏既有对象生命周期和序列化布局。
* 所有系统资源都要正确释放，包括文件描述符、锁、页固定状态和动态内存。
* 涉及缓冲池、记录、索引、事务等状态变更时，保证元数据与实际数据同步。
* 涉及比较、类型转换和浮点数时，严格遵循题目、现有类型系统和输出格式。
* 涉及 SQL 时，非法语法、非法表名、非法列名、类型不匹配等行为必须符合 `labx.md`，不能静默成功。
* 不为通过单个样例而改变全局语义。
* 不进行与实验无关的全仓库格式化。

### 4.4 前置实验依赖

若当前实验依赖前置实验：

* 先确认 `rmdb/` 是否已经包含前置功能；
* 优先在 `rmdb/` 内补齐依赖，不直接修改已完成的 `lab1/`、`lab2/` 等历史交付目录；
* 完成后再将 `rmdb/` 同步为当前实验的 `labx/labx/` 交付源码目录；
* 不得让修复当前实验的操作破坏已经通过的前置实验测试。

## 5. 构建与测试

### 5.1 确定正确命令

先检查：

* `README.md`；
* `docs/` 中与当前实验相关的项目结构、测试说明和使用文档；
* `CMakeLists.txt`；
* `Makefile`；
* `test/`、`tests/`、`unit_test.cpp`；
* 已有脚本和 CI 配置。

使用仓库原有方式构建，不擅自更换构建系统。典型流程仅供参考：

```bash
cd rmdb
mkdir -p build
cd build
cmake ..
cmake --build . -j
ctest --output-on-failure
```

实际命令必须以仓库内容为准。

### 5.2 测试层次

至少完成以下验证：

1. **编译验证**：完整清理后仍能成功构建；
2. **官方测试**：运行题目提供的单元测试、SQL 测试或评分脚本；
3. **针对性自测**：覆盖本次修改的正常路径、边界条件和失败路径；


对于 SQL 类实验，还要核对：

* 列顺序；
* 行顺序是否有题目要求；
* 表头和分隔格式；
* 浮点数精度；
* 非法 SQL 是否按要求输出 `failure`；
* 输出文件是否位于题目要求的位置。

### 5.3 测试真实性

* 只有实际执行并通过的测试，才能写“通过”。
* 未运行的测试写“未运行”，并说明原因。
* 失败测试必须记录失败命令、错误摘要和当前处理结果。
* 不得删除构建错误、警告或失败记录后声称完成。
* 若环境缺少依赖、权限或凭据，完成所有可完成步骤，并明确记录阻塞项和复现命令。

## 7. 清理与交付目录

完成当前实验实现和测试后，先将 `rmdb/` 同步为当前实验的交付源码目录：

```bash
mkdir -p labx/labx
rsync -a --delete \
  --exclude 'build/' \
  --exclude 'cmake-build-*/' \
  --exclude '.git/' \
  --exclude '.github/' \
  --exclude '*.o' \
  --exclude '*.a' \
  --exclude '*.so' \
  --exclude '*.out' \
  --exclude '*.log' \
  --exclude '*.tmp' \
  --exclude '*.swp' \
  --exclude 'core' \
  --exclude 'core.*' \
  --exclude '*.zip' \
  --exclude '.DS_Store' \
  --exclude '__pycache__/' \
  rmdb/ labx/labx/
```

最终必须保留：

```text
labx/
├── labx.md
├── labx/                  # 修改后的完整源代码目录，必须保留
├── labx.zip               # 只包含应提交的源代码
└── 实验完成流程.md
```

清理源代码目录中的非交付产物，包括但不限于：

* `build/`、`cmake-build-*/`；
* 可执行文件、目标文件和静态/动态库；
* 测试临时数据库、日志、core dump；
* `.DS_Store`、编辑器缓存、交换文件；
* `.pytest_cache/`、`__pycache__/`；
* 旧的 zip 包或嵌套压缩包。

但不要删除仓库原本就跟踪、且项目运行所必需的文件。

## 8. ZIP 打包规则

### 8.1 基本要求

最终压缩包必须命名为：

```text
labx/labx.zip
```

必须满足：

* 不包含 `labx.md`；
* 不包含 `实验完成流程.md`；
* 不包含 `.git/`、`.github/`、构建目录、缓存、日志和临时文件；
* 不包含另一个 zip；
* 只包含完成该实验所需提交的源代码、头文件、构建配置及必要资源；
* 源代码目录 `labx/labx/` 仍然保留，不能打包后删除；
* 每次代码变化后必须重新生成 zip，不能保留过期压缩包。

### 8.2 路径层级

先检查 Lab1 的压缩结构：

```bash
unzip -l lab1/lab1.zip
```

当前实验必须与 Lab1 的压缩包根目录层级保持一致。

若 Lab1 范例不可用，默认从源代码目录内部打包，使压缩包解压后直接出现源码内容，而不是出现多余的外层 `labx/labx/`：

```bash
rm -f labx/labx.zip
(
  cd labx/labx
  zip -r ../labx.zip . \
    -x 'build/*' 'cmake-build-*/*' '.git/*' '.github/*' \
       '*.o' '*.a' '*.so' '*.out' '*.log' '*.tmp' '*.swp' \
       'core' 'core.*' '*.zip' '.DS_Store' '__pycache__/*'
)
```

如系统没有 `zip`，可以使用 Python 生成标准 ZIP，但排除规则必须相同。

### 8.3 打包后强制校验

必须执行：

```bash
unzip -t labx/labx.zip
unzip -l labx/labx.zip
```

并检查：

```bash
unzip -Z1 labx/labx.zip | grep -E '(^|/)(labx\.md|实验完成流程\.md|\.git|build|cmake-build-|__pycache__)(/|$)|\.zip$'
```

上面的检查应当没有输出。若有输出，删除压缩包、修正排除规则并重新打包。

还要确认压缩包不是空包，并包含关键源码和构建文件。

## 9. Git 协作、提交与推送

### 9.1 协作安全

* 不覆盖其他成员的修改。
* 不使用 `git push --force` 或 `--force-with-lease`。
* 不擅自改写已有提交历史。
* 不提交凭据、Token、SSH 私钥、本地配置或大型无关文件。
* 不提交构建目录和临时产物。
* 合并冲突时，不凭猜测删除对方代码；应理解双方改动后完成最小合并。

### 9.2 提交前检查

执行：

```bash
git status --short
git diff --check
git diff -- rmdb labx
git diff --stat
```

确认：

* 只有预期文件被修改；
* 没有冲突标记；
* 没有明显尾随空格或格式错误；
* `实验完成流程.md` 与实际改动一致；
* `labx.zip` 是最新版本并通过校验。

### 9.3 暂存与提交

只暂存目标实验相关内容：

```bash
git add rmdb labx/labx labx/labx.zip labx/实验完成流程.md
```

若本次任务还明确要求修改根目录文件，再单独添加，不要使用无差别的 `git add .`。

推荐提交信息：

```text
labx: complete assignment requirements
```

或使用更具体的格式：

```text
lab2: implement query execution and DML operators
```

提交前再次查看：

```bash
git diff --cached --stat
git diff --cached --check
```

提交后记录 commit：

```bash
git commit -m "labx: complete assignment requirements"
git rev-parse --short HEAD
```

### 9.4 推送远程仓库

确认当前分支和远程跟踪关系后推送：

```bash
git branch --show-current
git remote -v
git push
```

若当前分支尚未设置 upstream：

```bash
git push -u origin "$(git branch --show-current)"
```

推送失败时：

* 不强推；
* 记录完整错误摘要；
* 若远程有新提交，先 `git fetch`，检查差异并安全合并或 rebase；
* 若缺少网络、权限或凭据，在 `实验完成流程.md` 中写明“代码已提交但未成功推送”，并给出待执行命令；
* 不得在未成功推送时声称已推送。

推送成功后更新 `实验完成流程.md` 中的 commit 和推送状态；如该更新产生新改动，应再提交并推送，确保远程仓库中的报告也是最终版本。

## 10. 完成定义

只有同时满足以下条件，才能宣布某个 Lab 完成：

* [ ] 已完整阅读并逐项覆盖 `labx.md`；
* [ ] 所有要求的接口和功能均已实现；
* [ ] `rmdb/` 保存当前最新实现；
* [ ] 没有通过修改测试、硬编码输出或绕过逻辑作弊；
* [ ] 项目可从干净构建目录成功编译；
* [ ] 已运行适用的官方测试和补充测试；
* [ ] 测试结果被真实记录；
* [ ] `labx/实验完成流程.md` 已完成且与实际修改一致；
* [ ] `labx/labx/` 源代码目录仍然存在；
* [ ] `labx/labx.zip` 已重新生成；
* [ ] ZIP 不包含 `labx.md`、实验报告、构建产物或嵌套 ZIP；
* [ ] `unzip -t` 校验成功；
* [ ] Git 仅提交了预期文件；
* [ ] 已创建清晰的 commit；
* [ ] 已成功推送到远程仓库，或真实记录无法推送的原因。
